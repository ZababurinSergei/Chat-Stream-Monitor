## 🎯 Описание

**AST Analyzer** — это мощный инструмент статического анализа кода, специально разработанный для оптимизации взаимодействия с большими языковыми моделями (ChatGPT, Claude, Gemini и др.).

### Ключевые возможности

| Возможность | Описание | Экономия |
|-------------|----------|----------|
| 🔍 **AST-парсинг** | Глубокий анализ структуры кода | - |
| ✂️ **Сжатие кода** | Удаление реализации, сохранение сигнатур | до **90%** токенов |
| 🎒 **Smart-контекст** | Автоматический сбор связанных файлов | до **70%** контекста |
| 💥 **Impact Analysis** | Оценка последствий изменений | время анализа |
| 🗑️ **Dead Code** | Поиск неиспользуемого кода | чистота кода |
| 📊 **Визуализация** | Интерактивные графы зависимостей | понимание архитектуры |

### Поддерживаемые форматы

- ✅ **JavaScript** (.js, .mjs, .cjs)
- ✅ **TypeScript** (.ts, .tsx)
- ✅ **Vue.js** (.vue) - извлекает script блоки
- ✅ **JSX** (.jsx)

## 📖 Режимы работы

### 1. project - Граф зависимостей проекта

**Назначение:** Построение интерактивной карты зависимостей всего проекта от корневого файла.

**Выходные файлы:**
- `report.html` - Интерактивный граф с возможностью zoom/pan
- `output.svg` - Векторный граф зависимостей
- `output.json` - Структура графа в JSON
- `output.dot` - Исходный код на языке DOT

**Команда:**
```bash
node graph-analyzer.js project <путь_к_файлу> [глубина]

# Глубина (optional): число или 'all' (по умолчанию Infinity)
```

**Примеры:**
```bash
# Полный граф (все зависимости)
node graph-analyzer.js project ./src/index.js

# Ограничение глубиной 2 уровня
node graph-analyzer.js project ./src/index.js 2

# Без ограничений (явно)
node graph-analyzer.js project ./src/index.js all
```

**Визуализация:**
- 🔵 **Синий узел** - входной/корневой файл
- 🟣 **Фиолетовые узлы** - обычные модули
- 🔴 **Красные пунктирные стрелки** - циклические зависимости

---

### 2. file - Внутренний граф файла

**Назначение:** Анализ внутренних связей внутри одного файла (вызовы функций, использование переменных).

**Выходные файлы:** (те же, что и в режиме project)

**Команда:**
```bash
node graph-analyzer.js file <путь_к_файлу>
```

**Пример:**
```bash
node graph-analyzer.js file ./src/services/api.js
```

**Что покажет граф:**
- Связи между функциями внутри файла
- Зависимости между переменными/константами
- Экспортируемые сущности

---

### 3. minify - Сжатие кода для ИИ

**Назначение:** Максимальное сжатие исходного кода с сохранением всей важной информации для ИИ.

**Что удаляется:**
- ❌ Тела функций (остаются сигнатуры)
- ❌ Логика вычислений
- ❌ Циклы и условия
- ❌ Внутренние переменные

**Что сохраняется:**
- ✅ Импорты/экспорты
- ✅ Сигнатуры функций (название, параметры, тип возврата)
- ✅ JSDoc комментарии
- ✅ TypeScript интерфейсы и типы
- ✅ Экспортируемые константы (без значений)

**Экономия токенов:** до **90%** по сравнению с исходным кодом

**Команда:**
```bash
node graph-analyzer.js minify <путь_к_файлу>
```

**Выход:** `ai-context.txt`

**Пример:**
```bash
node graph-analyzer.js minify ./src/components/UserProfile.tsx
```

**Исходный код:**
```typescript
export function calculateTotal(items: Item[]): number {
    let total = 0;
    for (const item of items) {
        total += item.price * item.quantity;
        if (item.discount) {
            total -= total * item.discount;
        }
    }
    return total;
}
```

**Результат (ai-context.txt):**
```typescript
export function calculateTotal(items: Item[]): number { /* реализация скрыта */ }
```

### 4. prompt-pack - Сборка контекста для ИИ

**Назначение:** Автоматическая сборка оптимального контекста для передачи ИИ при работе над конкретным файлом.

**Логика работы:**
1. Анализирует целевой файл и находит все его зависимости
2. Целевой файл включается **полностью**
3. Зависимости включаются в **сжатом виде** (minify)
4. Генерируется Markdown файл с готовым системным промптом

**Команда:**
```bash
node graph-analyzer.js prompt-pack <путь_к_файлу> [глубина]

# Глубина (default: 2) - как глубоко искать зависимости
```

**Выход:** `ai-prompt-bundle.md`

**Пример:**
```bash
node graph-analyzer.js prompt-pack ./src/checkout/OrderService.ts 2
```

**Структура выходного файла:**
```markdown
# 🧠 КОНТЕКСТ ДЛЯ ИИ АССИСТЕНТА

## 📋 ИНСТРУКЦИЯ ДЛЯ ИИ:
[Системный промпт с инструкциями]

---

## 🎯 ЦЕЛЕВОЙ ФАЙЛ
[Полный код OrderService.ts]

---

## 🔗 ЗАВИСИМОСТИ ПРОЕКТА
[Сжатые версии всех файлов, от которых зависит OrderService]
```

### 5. impact - Анализ зоны влияния

**Назначение:** Оценка того, какие файлы и функции будут затронуты при изменении или удалении конкретной сущности.

**Команда:**
```bash
node graph-analyzer.js impact <путь_к_файлу> <имя_сущности>
```

**Выход:** `ai-impact-report.md`

**Пример:**
```bash
node graph-analyzer.js impact ./src/database/queries.ts findUserById
```

**Что анализируется:**
- Импортирует ли какой-либо файл указанную сущность
- В каких функциях/методах она используется
- Контекст использования (топ-уровень или внутри функций)

**Пример отчета:**
```markdown
# ⚠️ ОТЧЕТ ПО ЗОНЕ ВЛИЯНИЯ ИЗМЕНЕНИЙ

**Цель:** Изменение/удаление сущности `findUserById` в файле `src/database/queries.ts`

🚨 **ОБНАРУЖЕНО 3 ЗАВИСИМЫХ ФАЙЛА!**

### 1. `src/api/users.ts`
Используется в:
  - [ ] Внутри `function getUserHandler()`
  - [ ] Внутри `arrow function updateUser()`

### 2. `src/auth/session.ts`
Используется в:
  - [ ] Внутри `function validateSession()`
```

### 6. dead-code - Поиск мертвого кода

**Назначение:** Выявление неиспользуемого кода для безопасного удаления.

**Два уровня анализа:**

1. **Локальный** (внутри файла)
    - Переменные, объявленные но не использованные
    - Функции, объявленные но не вызванные

2. **Глобальный** (по всему проекту)
    - Экспортируемые сущности, которые никто не импортирует

**Команда:**
```bash
node graph-analyzer.js dead-code <путь_к_файлу>
```

**Выход:** `ai-dead-code-report.md`

**Пример:**
```bash
node graph-analyzer.js dead-code ./src/legacy/utils.ts
```

**Пример отчета:**
```markdown
# 🗑️ ОТЧЕТ ПО НЕИСПОЛЬЗУЕМОМУ КОДУ

**Анализируемый файл:** `src/legacy/utils.ts`

## 🚫 Внутренний мертвый код
*Локальные сущности, объявленные, но не используемые в файле:*

  - [ ] `tempCounter`
  - [ ] `parseDeprecatedFormat`

## 📦 Бесполезные экспорты
*Экспортируются, но не импортируются нигде в проекте:*

  - [ ] `export oldCalculateMethod`
  - [ ] `export LEGACY_CONFIG`
```

## 🤖 Промпты для ИИ

### Промпт для анализа графа (режим project)

```markdown
## 📋 Инструкция для ИИ

Я загружаю граф зависимостей моего проекта в формате JSON.

**Задача:** Проанализируй архитектуру и предложи:

1. Основные модули и их ответственность
2. Потенциальные проблемы (циклические зависимости)
3. Рекомендации по улучшению структуры

**Данные:** [Вставьте содержимое output.json]
```

### Промпт для рефакторинга (режим prompt-pack)

```markdown
## 📋 Инструкция для ИИ

Я загружаю контекст для рефакторинга файла `[target-file]`.

**Целевой файл:** дан полностью
**Зависимости:** даны в сжатом виде (только сигнатуры)

**Задачи:**
1. [Опишите вашу конкретную задачу, например:]
   - Добавьте кэширование в функцию getUserData
   - Перепишите calculateTotal с использованием reduce
   - Добавьте обработку ошибок

**Ограничения:**
- Не изменяйте сигнатуры экспортируемых функций
- Сохраняйте обратную совместимость с зависимостями
- Используйте типы из предоставленных интерфейсов
```

### Промпт для Impact Analysis (режим impact)

```markdown
## 📋 Инструкция для ИИ

Я планирую изменить функцию `[entity-name]` в файле `[file-path]`.

**Отчет о влиянии:** [вставьте ai-impact-report.md]

**Задача:**
Для каждого файла из отчета:
1. Объясни, как изменение затронет этот файл
2. Предложи конкретные правки для адаптации
3. Оцени сложность изменений (low/medium/high)

**Дополнительно:** Сгенерируй исправленные версии файлов
```

### Промпт для Dead Code (режим dead-code)

```markdown
## 📋 Инструкция для ИИ

Я анализирую файл `[file-path]` на наличие мертвого кода.

**Отчет:** [вставьте ai-dead-code-report.md]

**Задачи:**
1. Для локального мертвого кода: предложи безопасное удаление
2. Для бесполезных экспортов: 
   - Напиши, как переделать в приватные сущности
   - Или аргументируй почему их можно удалить

**Результат:** Предоставь исправленную версию файла
```

### Универсальный промпт для работы с minify

```markdown
## 📋 Инструкция для ИИ

Я предоставляю сжатую версию файла (только сигнатуры без реализации).

**Файл:** [вставьте ai-context.txt]

**Задача:** Напиши новый модуль/компонент, который использует API из этого файла.

**Требования:**
- Импортируй все необходимые типы и функции
- Используй предоставленные интерфейсы
- Следуй существующим паттернам именования
```

## 💡 Советы и рекомендации

### 1. Экономия токенов

| Режим | Размер относительно исходного | Когда использовать |
|-------|------------------------------|--------------------|
| Исходный код | 100% | - |
| `minify` | 10-30% | ⭐ Для большинства задач с ИИ |
| `prompt-pack` | 15-40% | При работе над конкретным файлом |
| `prompt-pack` + глубина 1 | 20-50% | Для изолированных изменений |

### 2. Выбор глубины для prompt-pack

```bash
# Глубина 1 - только прямые зависимости (быстро, дешево)
node graph-analyzer.js prompt-pack ./src/file.js 1

# Глубина 2 - зависимости зависимостей (баланс)
node graph-analyzer.js prompt-pack ./src/file.js 2

# Глубина 3+ - максимальный контекст (медленно, дорого)
node graph-analyzer.js prompt-pack ./src/file.js 3
```

### 3. Интеграция в CI/CD

```yaml
# .github/workflows/analyze.yml
name: AST Analysis
on: [pull_request]
jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm install
      - run: node graph-analyzer.js impact ./src/changed-file.ts changedFunction
      - run: cat ai-impact-report.md >> $GITHUB_STEP_SUMMARY
```

### 4. Лучшие практики

- ✅ **Всегда используйте `impact`** перед удалением публичных API
- ✅ **Периодически запускайте `dead-code`** для поддержания чистоты
- ✅ **Используйте `minify`** для быстрых вопросов к ИИ о файле
- ✅ **Начинайте с глубины 2** для `prompt-pack`, увеличивайте если не хватает
- ❌ **Не отправляйте полные файлы** если можно использовать `minify`
- ❌ **Не удаляйте экспорты** без предварительного `impact` анализа
